# rfs-v2-hydroviewer

## Project Overview
- Vanilla JS + Vite 6 — single-page dashboard for past/current/forecast river conditions globally, driven by the GEOGloWS River Forecast System (RFS)
- ArcGIS Maps SDK + Calcite Components for the map UI; Plotly for time-series + report charts; zarrita for fetching RFS Zarr stores; Materialize CSS for the navbar / modals / tooltips / dropdowns; i18n via a custom `intl.js` (English / Spanish / French)
- Authenticates against Supabase Auth via `@aquaveo/geoglows-auth/core` (no React in this app — it consumes the lib's vanilla `core` surface, not `react`). Cross-app SSO with apps.geoglows + aquiferx + grace is automatic via the shared Supabase project + same origin (when proxied through the portal at `/hydroviewer/`)
- **Two parallel deployments**: `rfs-v2-hydroviewer.vercel.app` (Vercel project, served via the portal rewrite at `portal-dev.geoglows.org/hydroviewer/`) and `hydroviewer.geoglows.org` / `app.riverforecastsystem.com` (GitHub Pages, older — see `CNAME`). The portal-canonical access path is the Vercel deployment via the rewrite; the GH Pages deployment exists from a prior era and is a separate operational decision to consolidate

## Architecture
- **Sub-app of the GEOGloWS portal**: integrated via three rewrite rules in `apps.geoglows/vercel.json` (`/hydroviewer`, `/hydroviewer/`, `/hydroviewer/:path+`). Cataloged as `rfs` in `apps.geoglows/apps.json`
- **Three deployment contexts** handled by `vite.config.js`:
  - **Local dev** — `npm run dev`, assets at `/assets/...`
  - **Vercel preview** — autogenerated subdomain, `isVercelNonProd` guard forces `base: '/'`, assets at `/assets/...`
  - **Production via portal** — `VITE_BASE_PATH=/hydroviewer/` produces `/hydroviewer/assets/...` so portal-proxied URLs resolve
- **Auth listener registration**: `src/auth-bootstrap.js` registers `supabase.auth.onAuthStateChange("INITIAL_SESSION", ...)` immediately on module load. `src/main.js` imports it FIRST. rfs's `main.js` has no top-level awaits today, so the strict ordering grace's auth-bootstrap requires (listener-before-zarr-awaits) is not load-bearing here — but the FIRST-import convention protects against future additions of top-level awaits silently breaking SSO. The lib's 2s safety-net timeout backstops the listener
- **Surgical auth-slot rendering**: `renderAuthAction(state)` returns an HTML string; only the `<div id="auth-action">` slot's `innerHTML` is replaced on auth-state changes. The surrounding `.nav-bar-wrapper` and `<arcgis-map>` are NEVER re-rendered — ArcGIS map components AND Materialize's dropdown initializations (`M.Dropdown.init`, etc.) must not be torn down

## Key Files
- `index.html` — single-page entry; `<div id="auth-action" class="nav-auth">` slot is the third child of `.nav-bar-wrapper` (after the `<h4>` title and the `.nav-buttons` cluster)
- `src/main.js` — imports `./auth-bootstrap.js` as the **first line**; then CSS, then domain modules
- `src/auth-bootstrap.js` — Supabase client + auth adapter + `mountSignInModal` + `#auth-action` slot updater. Module-level `authState` + surgical innerHTML
- `src/ui.js` / `src/intl.js` / `src/data/main.js` / `src/bookmarks.js` / `src/states/state.js` / `src/plots.js` / `src/map.js` / `src/reports.js` — domain logic for UI controls, i18n, RFS data fetching/caching, saved-river bookmarks, app state, time-series + reports plots, and the ArcGIS map setup
- `src/css/main.css` — primary stylesheet; `.nav-bar-wrapper` uses `margin-left: auto` on `.nav-buttons` so buttons + auth slot stay grouped on the right (h4 stays left)
- `src/css/materialize.overrides.css` — Materialize CSS adjustments for this app's look
- `src/css/report.print.css` — print-only styles for the Reports modal
- `vite.config.js` — function-with-env signature; `loadEnv` + `isVercelNonProd` guard. Preserves the `injectAppVersionDate` plugin (replaces `%APP_VERSION_DATE%` placeholder in index.html), `worker: { format: 'es' }`, `build: { target: 'esnext' }`, and `define: { global: 'globalThis' }`
- `vercel.json` — `installCommand: "npm install --legacy-peer-deps"` (the lib has React 19 peer deps that vanilla rfs doesn't satisfy), no rewrites
- `.npmrc` — `legacy-peer-deps=true` so local `npm install` matches Vercel
- `CNAME` — GitHub Pages custom domains (`hydroviewer.geoglows.org`, `app.riverforecastsystem.com`); not used by the Vercel deployment

## Conventions
- Vanilla JS only (no TypeScript, no JSX, no React)
- HTML escape discipline: any `${value}` interpolation in template-string-then-innerHTML rendering MUST go through `escapeHtml()` from `@aquaveo/geoglows-auth/core`. Currently the only HTML rendering rfs's auth code owns is the auth-slot, which uses the lib's pre-escaped output. Any future addition that interpolates user/API data into innerHTML must use `escapeHtml()`
- The lib owns the stable element IDs: `#geoglowsSignIn`, `#geoglowsSignOut`, `#geoglowsAuthActionAvatar`. Bind to those exact IDs in event handlers
- `mountSignInModal` is mounted ONCE at module load and bridged via the `geoglows:sign-in-requested` window event so any UI surface can request sign-in without holding a direct modal reference
- Materialize CSS is initialized via `M.AutoInit()` plus targeted `M.Dropdown.init(...)` for the language picker. The auth-slot uses semantic class names (`.geoglows-auth-action-*`) that don't collide with Materialize's `.btn` etc.
- Both Vercel and GH Pages deployments exist; the **Vercel deployment is the canonical portal-integrated path** (served via `portal-dev.geoglows.org/hydroviewer/`). The GH Pages site (`hydroviewer.geoglows.org`) is a parallel older deploy — `VITE_BASE_PATH` only affects the Vercel build

## Commands
- `npm install --legacy-peer-deps` — install deps (the `--legacy-peer-deps` is also defaulted via `.npmrc`)
- `npm run dev` — Vite dev server
- `npm run build` — production build. For the production-styled build use `VITE_BASE_PATH=/hydroviewer/ npm run build`
- `npm run preview` — serve the production build locally

## Environment
- `VITE_SUPABASE_URL`, `VITE_SUPABASE_PUBLISHABLE_KEY` — required for any environment that signs users in. Set per-environment on Vercel (Production + Preview + Development). Setting only Production silently breaks Preview deploys
- `VITE_BASE_PATH` — set to `/hydroviewer/` on Vercel **Production only**. Preview is handled by the `isVercelNonProd` guard, which forces `base: '/'` regardless of this var. Local dev leaves it unset (also `'/'`)
- The Supabase URL + publishable key are shared across apps.geoglows + aquiferx + grace + rfs — same project, same keys. Cross-app SSO falls out of having all four apps on the same origin (after the portal rewrites) reading the same `localStorage` token
